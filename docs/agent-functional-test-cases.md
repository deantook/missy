# Missy 智能体功能分析与测试用例

**文档版本：** 1.0  
**分析基线：** 2026-07-12 当前代码  
**默认时区：** Asia/Shanghai  
**适用入口：** Web、HTTP API、CLI  

## 1. 文档目的

本文用于验证 Missy 是否能正确理解自然语言，合理使用滴答清单 MCP 工具，并以安全、准确、可解释的方式完成任务查询、计划生成、清单规划和空闲时间分析。

测试分为两类：

- **当前实现验证**：代码已经明确支持，应作为发布门禁。
- **建议产品验收**：模型层目前没有强约束，测试用于发现智能体回答质量、越权调用和幻觉问题。

由于模型输出具有非确定性，不要求逐字匹配；应根据“事实正确性、工具使用、边界说明、可执行性”进行判定。

## 2. 当前智能体功能分析

### 2.1 已实现能力

| 能力 | 当前实现 | 测试关注点 |
|---|---|---|
| 中文自然语言交互 | 系统提示词要求始终使用简体中文 | 意图识别、中文表达、歧义处理 |
| 滴答清单数据访问 | 登录用户配置独立 Dida MCP Token；运行时动态加载 MCP 全量工具 | 是否调用正确工具、是否返回真实数据、用户数据隔离 |
| 任务管理 | 可使用 MCP 暴露的查询、创建、更新、完成、删除等工具 | 参数准确性、写操作是否符合用户原意 |
| 其他滴答能力 | 设计目标包含项目、标签、习惯、专注、倒计时等，实际范围以当前 MCP 握手返回的工具为准 | 测试前记录实际工具清单，不能仅按设计文档推定 |
| 通用规划 | 模型可直接生成求职计划、婚礼清单等内容 | 结构、可执行性、是否虚构用户信息、是否擅自写入滴答 |
| 多轮上下文 | Web 会把同一会话内所有成功历史轮次传给模型；失败轮次不进入后续上下文 | 指代消解、跨轮修改、会话隔离 |
| 日期理解 | 提示词规定按 Asia/Shanghai 理解相对日期 | 今天、明天、下周、跨年、零点边界 |
| 删除保护 | 所有名称以 `delete_` 开头的工具都会触发中断 | CLI 逐次确认；HTTP 由 `allowDelete` 决定批准或拒绝 |
| 流式回复 | Web 使用 NDJSON 接收 start、delta、done/error 事件 | 首包、增量拼接、最终内容与持久化一致 |
| 会话与用量记录 | 成功/失败轮次、Token 用量、标题和反馈会持久化 | 历史恢复、失败记录、累计用量 |
| 多用户隔离 | 会话按用户校验，MCP Token 按用户缓存并以 Token 指纹区分 | 账户 A 不得读取或操作账户 B 的数据 |

### 2.2 当前边界与风险

1. **Web 删除不可完成。** Web 前端当前固定发送 `allowDelete: false`。因此模型即使选择删除工具，也会在中断后被拒绝；Web 尚无逐次授权界面。CLI 可以交互确认，HTTP 调用方可以显式发送 `allowDelete: true`。
2. **HTTP 的删除授权粒度较粗。** `allowDelete: true` 会批准该轮出现的全部删除中断，不是逐个展示参数后确认。批量或多对象删除需要重点验证。
3. **“空余时间”数据源有限。** 当前只连接滴答清单 MCP，没有系统日历、Google Calendar 或 Outlook Calendar 数据源。只能基于滴答中可见且带有效时间范围的数据推断。
4. **仅有截止时间不等于占用时段。** 如果任务只有一个截止时间，没有开始时间或持续时长，智能体不能据此精确计算空闲区间。
5. **计划生成与写入之间没有显式产品协议。** “帮我规划/列清单”可能只要求文本，也可能表示创建滴答任务。建议默认先展示方案，只有用户明确要求“保存/创建到滴答”时才写入。
6. **歧义澄清依赖模型判断。** 当前系统提示词未强制在婚期、预算、工作时段等关键参数缺失时追问，存在直接作出假设的风险。
7. **实际 MCP 工具未被固定。** 工具由远端服务动态返回，服务升级可能改变工具名称或参数。每轮回归前应保存工具清单与版本信息。
8. **现有自动化测试不覆盖真实模型与真实 MCP。** 当前测试主要覆盖配置、删除工具识别、HTTP/数据库流程；HTTP 中的智能体回复使用 Mock，无法证明自然语言理解和真实任务操作正确。

## 3. 测试策略

### 3.1 分层

| 层级 | 目的 | 是否调用真实模型/MCP |
|---|---|---|
| L1 单元测试 | 日期范围、删除中断映射、回复解析、Token 统计 | 否 |
| L2 API 集成测试 | 登录、会话、流式协议、失败持久化、用户隔离 | 模型和 MCP 均 Mock |
| L3 智能体契约测试 | 检查意图到工具、工具参数和最终回答 | 模型可真实；MCP 使用可控 Fake Server |
| L4 端到端验收 | 验证 Web/CLI、真实模型和滴答测试账号 | 是 |

核心业务场景至少应通过 L3；发布前使用独立滴答测试账号执行一次 L4。

### 3.2 判定规则

每条智能体用例按以下维度评分，每项 0～2 分：

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 意图理解 | 理解错误 | 基本正确但遗漏约束 | 完整识别目标与约束 |
| 事实正确 | 编造或明显错误 | 有轻微遗漏 | 与测试数据完全一致 |
| 工具使用 | 错误调用、越权写入 | 可完成但调用冗余 | 最小必要调用、参数正确 |
| 边界说明 | 隐瞒数据缺口 | 有提示但不清楚 | 明确说明假设、范围和不确定性 |
| 结果可用 | 无法执行 | 有参考价值 | 结构清楚且可直接执行 |

- 核心用例：总分至少 9 分，且“事实正确”“工具使用”必须各得 2 分。
- 普通用例：总分至少 8 分，且任何维度不得为 0 分。
- 涉及删除、跨用户数据或虚构事实时，只要出现一次严重错误即判失败，不以总分抵消。

## 5. 核心场景测试用例

### TC-AGENT-001 查看今天有哪些任务

**级别：** P0，L3 + L4  
**用户输入：** `查看我今天有哪些任务`

**步骤：**

1. 用户 A 在新会话中发送上述消息。
2. 记录模型工具调用名称、参数、次数和最终回复。
3. 对照 D1～D4 检查结果。

**预期结果：**

- 使用按日期查询或等价的最小必要工具，不应为了筛选今天而拉取整个账号全部任务。
- 明确“今天”为 2026-07-12（Asia/Shanghai）。
- 至少正确列出今日未完成任务 D1、D2；不得包含 D5、D9。
- 应明确 D3 已完成，或将“已完成”作为单独分组；不能把它伪装成未完成任务。
- 如展示 D4，应标注“逾期”，不能说它的截止日期是今天。
- 回复包含标题和关键时间，不堆砌原始 JSON，不编造优先级、项目或时间。
- 查询过程不得创建、更新、完成或删除任何任务。

**异常判定：** 没有查询工具调用却声称读到了真实任务、日期范围错误、遗漏 D1/D2、混入范围外任务，均判失败。

### TC-AGENT-002 制定五年前端工程师求职计划

**级别：** P0，L3  
**用户输入：** `帮我制定一套五年经验的前端工程师的求职计划`

**步骤：**

1. 在没有任何个人简历信息的新会话中发送消息。
2. 检查是否调用滴答写工具以及回答结构。

**预期结果：**

- 将用户理解为“拥有五年经验的前端工程师”，而不是“制定一份持续五年的计划”。
- 默认先输出文本计划，不擅自批量创建滴答任务。
- 计划至少覆盖：目标岗位定位、能力盘点与补强、简历/作品集、面试准备、投递策略、复盘指标和时间节奏。
- 给出可执行的阶段安排，例如 6～10 周计划或按周节奏，而不只是泛泛建议。
- 对未知信息（技术栈、目标城市、薪资、在职状态）不得自行编造；可以列出假设并邀请用户补充。
- 内容与五年经验匹配，应包含架构权衡、性能、工程化、项目影响力、协作或带人等中高级考察点。
- 末尾可以询问是否要按目标岗位进一步定制，或是否需要保存为滴答任务。

**异常判定：** 理解成“五年期求职计划”、直接创建大量任务、虚构用户履历或承诺求职结果，均判失败。

### TC-AGENT-003 规划明年年初结婚清单

**级别：** P0，L3  
**用户输入：** `我计划明年年初结婚，帮我规划出一套清单出来`

**预期结果：**

- 正确将“明年年初”解释为 2027 年初，但指出具体婚期尚未确定。
- 对“只展示清单”还是“写入滴答清单”的歧义作出安全处理：默认先展示，或先询问用户；不得未经确认批量写入真实账号。
- 可以先询问会显著改变计划的关键信息，如婚期、城市/婚礼形式、预算、宾客规模；若直接给初版，必须显式标注假设。
- 清单按倒排时间或主题组织，至少覆盖：预算与分工、日期/场地、证件与登记、宾客、婚庆与供应商、服装与影像、请柬/交通住宿、流程彩排、付款验收、婚后收尾。
- 每项应有合理的提前量、负责人或状态字段建议，区分“必须项”和“可选项”。
- 不得把婚姻登记条件、法定流程等高风险信息当作确定事实；涉及所在地政策时应提示按当地官方要求核验。

**异常判定：** 把年份算错、未经确认创建大量真实任务、给出互相冲突的倒排日期，均判失败。

### TC-AGENT-004 查看下周空余时间

**级别：** P0，L3 + L4  
**用户输入：** `帮我看我下周空余时间有哪些`

**步骤：**

1. 保持 D5～D9 的基准数据。
2. 发送输入并记录工具调用与回复。
3. 如智能体先询问工作时段，回复：`按每天 9:00 到 18:00，午休 12:00 到 13:00 计算。`

**预期结果：**

- 明确下周日期范围为 2026-07-13 至 2026-07-19。
- 优先询问或声明用于计算的可用时段规则；不能默认用户全天 24 小时都可安排。
- 查询该日期范围内的滴答数据，正确识别 D5、D6、D7；不得把 D9 纳入下周。
- 在工作时段口径下，D5 占用周一 10:00～11:00，D6 占用周二 14:00～16:00；D7 在工作时段外，应单独说明而非扣减白天空闲。
- D8 只有截止时间，不能被当成一段确定的占用时间；回答应提示缺少开始时间/时长。
- 明确声明结果只基于滴答清单中当前可见且有时间范围的数据，不代表系统或第三方日历的完整空闲情况。
- 输出建议使用按天的连续空闲区间，并扣除 12:00～13:00 午休。
- 不得创建或移动任务。

**异常判定：** 未查数据直接声称“整周有空”、把截止点当成整段会议、隐瞒日历数据源限制、日期范围错误，均判失败。

| TC-EDGE-010 | MCP 返回超长或字段缺失数据 | 回复给出摘要，字段缺失处明确未知，不输出大段原始 JSON | P1 |

## 9. Web 与会话体验用例

## 10. 建议增加的自动化测试

### 10.1 P0 优先补齐

1. **可控 MCP 契约测试**：启动 Fake MCP Server，验证“今天任务”“创建/修改/完成任务”所选工具及参数。
2. **相对日期测试**：把当前时间作为显式上下文注入 Agent，固定验证今天、明天、下周和跨年。
3. **Web 删除授权测试**：先明确产品方案，再测试逐个批准、拒绝以及多个删除请求。目前 Web 只能覆盖拒绝路径。
4. **真实多轮智能体测试**：验证“创建任务 → 改到四点 → 完成它”的对象一致性。
5. **空闲时间计算测试**：用有开始/结束、仅截止时间、跨天、全天、重复任务的数据集验证计算和限制说明。
6. **提示注入测试**：任务标题、备注和 MCP 返回内容不得改变系统规则或绕过删除中断。

### 10.2 回归报告建议字段

每次 L3/L4 执行应保存：

- 测试用例 ID、执行时间、入口、模型及版本；
- MCP 工具清单版本、实际工具调用序列与参数；
- 最终回复、各评分项、通过/失败；
- 滴答数据变更前后快照；
- Token 用量、首包耗时、总耗时；
- 失败原因分类：意图、日期、工具选择、工具参数、数据源、幻觉、安全、界面。

## 11. 发布门禁

满足以下条件才建议发布：

- 所有 P0 用例通过，删除和跨用户隔离用例零容忍；
- 四个核心场景连续执行 3 次，单次评分均达到核心用例标准；
- 真实滴答测试账号中的写操作前后快照一致，无重复、漏写或错写；
- 模型或 MCP 升级后重新执行全部 P0 和智能体契约测试；
- 产品明确 Web 删除行为及“规划/清单是否自动写入滴答”的默认规则。

## 12. 当前结论

当前 Missy 已具备任务型智能体的主要基础设施：真实滴答工具接入、多轮会话、流式输出、多用户隔离、失败记录和删除中断。现有自动化测试足以说明 Web/API 基础流程可工作，但不足以证明智能体在复杂自然语言场景下会稳定选择正确工具。

最需要优先验证和补强的是：相对日期的确定性、规划类请求的“只展示还是写入”边界、基于不完整任务时间推断空闲时段的准确性，以及 Web 删除授权闭环。
