# Missy 项目风险评估

## 1. 项目结构概览

Missy 当前由以下部分组成：

- `src/`：Node.js/Express 后端、认证、数据库、Agent、MCP 和对话服务。
- `web/`：当前主 React Web 前端。
- `web-vanilla/`：已归档但仍可运行的 Vanilla 前端。
- `src-tauri/`：Tauri 桌面端壳，复用 `web/` 前端并连接远程 HTTP API。
- `migrations/`：PostgreSQL 数据库迁移。
- `tests/`：后端、认证、前端协议和调试能力测试。
- `docs/superpowers/`：设计和实施计划文档。

系统核心链路为：

```text
React Web / Tauri
        |
        v
Express HTTP API
        |
        +--> PostgreSQL：用户、会话、对话、Token 用量
        |
        +--> LangChain / DeepAgents：模型调用
        |
        +--> Dida365 MCP：用户任务数据读写
```

## 2. 当前验证结果

- 测试：13 个测试文件、55 个测试全部通过。
- TypeScript 类型检查：通过。
- React Web 生产构建：通过。
- 尚未覆盖：并发请求、压力限制、真实 MCP 异常、生产安全配置、客户端断开后的资源取消。

## 3. 风险清单

标记说明：`[用户体验相关]` 表示该风险除了技术或安全影响外，也会直接影响用户完成任务、理解状态或恢复操作。

### 高风险

#### R-01：Dida365 Token 明文存储

`users.dida_mcp_token` 保存完整 Token，更新接口也直接写入数据库。

- 证据：`migrations/001_initial.sql:3-12`
- 证据：`src/http.ts:189-199`

数据库备份、运维账号、SQL 权限或日志泄露后，攻击者可以直接操作用户的滴答清单账号。

建议使用 KMS 或应用级加密保存 Token 密文，并增加 Token 轮换和失效机制。

#### R-02 `[用户体验相关]`：Bearer Token 保存在 `localStorage`

Web 前端将登录 Token 写入 `localStorage`，服务端会话有效期为 30 天。

- 证据：`web/src/api/auth-storage.ts:1-16`
- 证据：`src/auth.ts:42-49`

一旦发生 XSS、恶意第三方脚本或供应链攻击，攻击者可以读取并长期复用 Token。

建议 Web 端改用 `HttpOnly`、`Secure`、`SameSite` Cookie；若必须保留 Bearer Token，应缩短有效期并增加刷新 Token、设备会话管理和主动撤销。

#### R-03 `[用户体验相关]`：缺少登录、注册和模型调用限流

当前认证和消息接口未发现速率限制、账户锁定、并发配额或用户级 Token 配额。

- 证据：`src/http.ts:116-149`
- 证据：`src/http.ts:291-332`

可能导致密码暴力破解、模型费用失控、Dida365 API 配额耗尽和数据库连接池压力过大。

建议对认证接口按 IP 和账号限流，对消息接口增加用户级限流、并发上限、每日 Token 配额和超时控制。

#### R-04：Tauri CSP 关闭

Tauri 配置将 CSP 设置为 `null`，同时启用了外部链接打开能力。

- 证据：`src-tauri/tauri.conf.json:22-24`
- 证据：`src-tauri/capabilities/default.json:5-6`

桌面端 WebView 一旦被注入脚本，攻击面的影响范围会显著扩大。

建议配置严格 CSP，仅允许必要的 API、字体和资源域名，并限制可打开的外部 URL。

### 中高风险

#### R-05 `[用户体验相关]`：修改密码不会撤销既有会话

修改密码只更新密码哈希，不会删除用户已有的 `auth_sessions`。

- 证据：`src/http.ts:179-186`

如果旧 Bearer Token 已泄露，修改密码后仍可能继续使用最长 30 天。

建议改密、注销、修改 Dida365 Token 时撤销相关会话，或提供“退出所有设备”功能。

#### R-06：应用启动时自动执行数据库迁移

HTTP 服务启动前会连接数据库并执行迁移。

- 证据：`src/server.ts:9-11`
- 证据：`src/db.ts:13-37`

生产应用进程需要 DDL 权限；迁移失败可能阻止服务启动，多实例发布也会增加发布耦合。

建议将迁移放入独立发布步骤，应用运行账号只保留必要的 DML 权限。

#### R-07 `[用户体验相关]`：同一会话缺少并发控制

同一 `conversationId` 可以同时触发多次 Agent 运行，历史读取和写入之间没有锁、版本号或队列。

- 证据：`src/chat-service.ts:51-73`
- 证据：`src/agent-runtime.ts:102-104`

可能出现历史读取重复、回复顺序错乱、标题竞争、Token 用量累计异常，以及同一 LangGraph `thread_id` 被并发使用。

建议按会话增加互斥队列，或使用数据库行锁/乐观锁，并明确限制同一会话并发运行。

#### R-08 `[用户体验相关]`：客户端断开后模型调用可能继续运行

流式响应停止写数据后，没有将客户端断开信号传递给模型或 MCP 调用。

- 证据：`src/http.ts:305-331`

用户关闭页面后，后台任务可能继续消耗模型费用和 MCP 资源。

建议监听 `req.aborted` 和 `res.close`，使用 `AbortController` 取消 Agent、模型和 MCP 请求。

#### R-09：Debug 数据可能暴露用户任务内容

调试事件会返回工具调用参数、工具结果预览和推理相关信息，结果预览最多 2048 字符。

- 证据：`src/debug-events.ts:14-20`
- 证据：`src/debug-events.ts:77-97`

这些数据可能包含任务标题、备注、地址或其他个人信息，并可能被浏览器缓存、录屏或监控系统收集。

建议生产环境默认关闭 debug，或采用白名单字段、参数脱敏和结果脱敏策略。

#### R-10 `[用户体验相关]`：上游错误信息可能泄露内部细节

非 debug 模式下仍会将原始错误消息返回给客户端。

- 证据：`src/debug-events.ts:22-32`

MCP 或模型 SDK 错误可能包含 URL、请求细节、内部路径或供应商信息。

建议对外只返回稳定错误码和通用文案，详细错误仅写入受控服务端日志，并过滤凭据。

### 中风险

#### R-11 `[用户体验相关]`：MCP 缓存存在并发竞态

同一用户的 MCP 连接缓存、失效和重建没有按用户加锁。

- 证据：`src/agent-runtime.ts:29-53`
- 证据：`src/agent-runtime.ts:61-78`

并发请求或 Token 更新期间，可能创建多个连接，或者关闭仍在使用的旧连接。

建议为每个用户维护连接初始化 Promise/互斥锁，并增加连接上限和关闭状态管理。

#### R-12 `[用户体验相关]`：生产部署和运维能力不足

当前项目未发现 Docker、CI/CD、结构化日志、指标采集和告警配置。数据库还有带固定本地凭据的默认连接串。

- 证据：`src/config.ts:70-100`

可能造成部署不一致、默认配置误用和故障定位困难。

建议增加 CI 中的测试、类型检查、依赖审计和构建验证；生产环境强制显式配置 `DATABASE_URL`；补充请求 ID、模型耗时、Token 用量和 MCP 错误指标。

#### R-13 `[用户体验相关]`：React 和 Vanilla 两套前端长期并存

`web/` 是主 React 前端，`web-vanilla/` 虽已归档但仍可运行，并且两套前端都有相关测试和协议实现。

这会增加接口、Markdown、choice prompt、安全修复和交互行为不同步的风险。

建议明确 Vanilla 的生命周期：彻底移除，或标记为独立 legacy 版本并停止默认构建和测试。

#### R-14 `[用户体验相关]`：过期会话缺少定期清理

`auth_sessions` 有 `expires_at` 和索引，但未发现定期删除过期记录的任务。

- 证据：`migrations/001_initial.sql:15-24`

长期运行后会话表可能持续增长，影响存储和查询维护。

建议增加定时清理任务或数据库定期维护作业。

## 4. 已有安全措施

- SQL 查询使用参数化形式，未发现明显 SQL 注入点。
- 密码使用带随机盐的 `scrypt` 哈希。
- 服务端只保存会话 Token 的 SHA-256 哈希。
- 用户资源查询普遍包含 `user_id` 约束，已有越权测试。
- Markdown 渲染使用 DOMPurify。
- 已关闭 Express `x-powered-by`。
- MCP 连接支持空闲回收。
- 已有基础错误码、健康检查和流式响应测试。

## 5. 建议整改顺序

1. 加密存储 Dida365 Token，轮换生产环境中已有的 Token。
2. 增加认证、消息接口的限流、并发控制和模型费用配额。
3. 改用安全 Cookie 或完善短期 Bearer/刷新 Token 机制。
4. 修改密码和 Token 后撤销旧会话。
5. 增加会话级互斥和客户端断开取消机制。
6. 为 Tauri 设置严格 CSP，限制外部链接能力。
7. 关闭或脱敏生产 debug 数据，并统一错误返回策略。
8. 将数据库迁移从应用启动流程中分离，补充 CI/CD 和生产监控。
9. 决定 `web-vanilla/` 的最终去留，减少双前端维护面。
