# Missy 用户体验风险评估

## 1. 评估范围

本文聚焦会影响用户“看懂当前状态、完成操作、恢复失败、找回历史和持续使用”的风险。它不等同于视觉设计评审；部分问题由后端可靠性、数据一致性或安全策略引起，但最终会表现为用户体验问题。

风险等级：

- 高：用户可能丢失操作结果、无法继续使用，或误以为操作已完成。
- 中：主要流程仍可完成，但容易困惑、重复操作或降低信任。
- 低：影响特定设备、辅助技术或长期使用效率。

## 2. 风险清单

### UX-01 `[高]`：失败消息没有“重试”入口

发送失败后，前端只把临时消息标记为失败并显示错误文本，没有保留可直接重试的按钮；用户需要重新输入整条消息。

- 证据：`web/src/context/chat/sendMessage.ts:101-106`
- 证据：`web/src/components/MessageBubble.tsx:32-41`

模型超时、网络抖动或 MCP 暂时不可用时，用户很容易丢失较长的任务描述，并可能重复提交导致重复执行。

建议：为失败消息增加“重试”和“复制消息”操作；重试前区分“服务未开始执行”和“服务可能已执行但响应丢失”两种状态，避免重复创建任务。

### UX-02 `[高]`：创建新会话失败时没有用户反馈

新建会话请求失败时，`createConversation` 直接返回 `null`，调用方没有显示 Toast、错误页或恢复操作。

- 证据：`web/src/context/ChatContext.tsx:65-79`

用户点击发送后可能没有任何可见变化，不清楚是网络问题、服务不可用还是消息没有发送。

建议：显示明确错误提示，并提供“重试创建会话”；同时记录失败原因，避免静默吞错。

### UX-03 `[中高]`：初始历史加载失败会显示为空白状态

登录后加载会话列表使用了未处理的异步调用，失败时没有设置错误状态或展示重试入口。

- 证据：`web/src/context/ChatContext.tsx:191-203`
- 证据：`web/src/pages/ChatPage.tsx:55-68`

用户可能误以为历史会话为空，尤其是在网络暂时不可用时，容易产生“数据丢失”的误解。

建议区分“正在加载”“加载失败”“确实没有历史”三种状态，并提供刷新按钮。

### UX-04 `[中高]`：流式响应中断后的恢复能力不足

前端检测到流式响应没有 `done/error` 事件后只抛出通用错误，没有断点续传、重试或从服务端重新拉取本轮状态的流程。

- 证据：`web/src/api/stream.ts:29-38`
- 证据：`web/src/context/chat/sendMessage.ts:101-109`

用户可能已经看到部分回复，但最终无法判断任务是否已执行成功。直接重试又可能重复创建或修改任务。

建议为每轮引入稳定的 turn 状态查询接口和幂等键；断流后先查询服务端结果，再决定是否允许重试。

### UX-05 `[中]`：一次慢请求会锁住整个聊天操作区

发送期间 Composer 被禁用，历史回复的点赞/点踩也被禁用；长时间的模型或 MCP 调用会让用户无法继续输入、评价或切换操作。

- 证据：`web/src/components/Composer.tsx:10-14`
- 证据：`web/src/components/MessageBubble.tsx:43-68`

用户只能等待，无法取消当前请求，也无法执行与当前请求无关的浏览操作。

建议增加“停止生成”按钮；将反馈操作与发送状态解耦；允许用户查看历史会话，同时保留当前任务的运行状态。

### UX-06 `[中]`：选择弹窗关闭后可能重复出现

选择弹窗的关闭状态只保存在 `ChatPage` 的内存状态中。重新进入页面或重新加载会话后，最新一条包含 choice prompt 的回复可能再次弹窗。

- 证据：`web/src/pages/ChatPage.tsx:40-42`
- 证据：`web/src/pages/ChatPage.tsx:72-85`

用户已经选择“我自己输入”后，重新打开同一会话仍可能被同一个弹窗打断。

建议将已处理的 prompt 状态与 turn ID 持久化，或在消息展示层提供“再次打开”而不是自动弹出。

### UX-07 `[中]`：选择弹窗缺少完整的键盘焦点管理

弹窗打开时只将焦点放到关闭按钮，并监听 Escape；没有焦点陷阱，Tab 可能移到背景页面控件。

- 证据：`web/src/components/ChoiceDialog.tsx:35-42`
- 证据：`web/src/components/ChoiceDialog.tsx:73-85`

键盘用户和使用辅助技术的用户可能误操作背景页面，或无法明确当前操作上下文。

建议实现焦点陷阱、关闭后恢复原焦点，并补充键盘导航和屏幕阅读器测试。

### UX-08 `[中高]`：历史会话前端只加载第一页

后端支持 cursor 分页，但前端请求没有传 cursor，也没有加载更多逻辑。超过默认数量的会话可能无法在界面中找到。

- 证据：`src/http.ts:211-224`
- 证据：`web/src/context/ChatContext.tsx:55-59`

长期使用后，用户会误以为旧会话被删除或丢失。

建议实现滚动加载/“加载更多”、历史搜索和按时间筛选，并明确显示是否还有更多结果。

### UX-09 `[中]`：原生 `prompt` 导致交互体验不一致

重命名和账户删除使用浏览器原生 `window.prompt`，账户删除还通过普通文本输入框收集密码，不是应用内的密码输入组件。

- 证据：`web/src/context/ChatContext.tsx:97-101`
- 证据：`web/src/components/settings/AccountActions.tsx:48-58`

不同浏览器和 Tauri WebView 的样式、键盘行为和错误反馈不一致；删除账户时密码可能直接显示在原生输入框中。

建议统一使用应用内 Dialog，提供密码类型输入、明确的危险操作说明、校验错误和提交状态。

### UX-10 `[中]`：错误状态会混合“执行失败”和“结果未知”

当前失败消息统一显示“请求失败”，但流式连接中断时，服务端可能已经完成了 MCP 写操作，前端没有向用户区分“未执行”“执行失败”和“已执行但响应丢失”。

- 证据：`web/src/context/chat/sendMessage.ts:85-106`
- 证据：`src/chat-service.ts:73-80`

用户可能重复发送创建任务、删除任务或批量操作，造成重复数据或重复修改。

建议给每轮增加可查询状态：`pending`、`succeeded`、`failed`、`unknown`；对于 `unknown` 状态先提供“查询执行结果”，再提供重试。

### UX-11 `[中]`：连接配置失败的反馈不够可操作

Token 验证失败统一显示“无效或服务暂时不可用”，但没有区分 Token 错误、MCP 服务不可用、网络超时和服务配置错误。

- 证据：`src/http.ts:190-196`
- 证据：`web/src/components/settings/TokenForm.tsx:24-36`

用户无法判断是重新复制 Token、稍后重试，还是联系管理员。

建议保留稳定错误码并映射为用户可执行的提示，例如“Token 无效，请重新复制”“服务暂时不可用，请稍后重试”“网络连接失败，请检查网络”。

### UX-12 `[中]`：长时间运行期间缺少阶段性进度信息

系统虽然产生 debug 阶段事件，但普通用户默认看不到 MCP 连接、工具调用、校验和重试阶段。

- 证据：`src/http.ts:321-322`
- 证据：`web/src/context/chat/sendMessage.ts:80-83`

当一次请求持续较久时，用户只能看到打字动画，不知道系统是在查询、执行还是等待上游服务。

建议在不暴露内部细节的前提下展示简化状态，例如“正在连接滴答清单”“正在执行任务”“正在核对结果”，并为超长阶段提供取消入口。

### UX-13 `[低]`：移动端和小屏幕下可用性仍需真实设备验证

项目通过 CSS media query 处理移动布局，但主 Shell、固定高度、软键盘、弹窗和调试面板之间存在复杂组合。

- 证据：`web/src/components/AppShell.module.css:122-175`
- 证据：`web/src/components/ChoiceDialog.module.css:206-220`

在移动浏览器或 Tauri 小窗口中，软键盘可能遮挡 Composer，调试面板可能压缩消息区域，弹窗操作区也可能需要额外滚动。

建议针对 iOS Safari、Android Chrome、Tauri 小窗口和横竖屏进行真实设备验收，重点测试软键盘、滚动位置、弹窗焦点和发送按钮可见性。
